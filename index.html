Development Workflow

Jan 5th, 2014

The idea of ‘Outside in development’ and ‘Acceptance Testing’ stresses the idea of 1.Think first and 2.Then Code.

If you know exactly what you want it is much easier to find the code for it. If you are collaborating in a team or handing over the project to another person, the acceptance tests will help them pick up where you left.

If you want your code to be open-source or ready for hand-over your commits should reflect the workflow and one should find all the code for one feature together in one commit.

Still you need to know what you want to be able to do it. If you have no vague idea, playing around might be the first step. Then when you have an idea, trash it all and start all over again, with “clean” workflow and commit stream.

User Stories and Acceptance Criteria

Dec 13th, 2013

Launch Academy uses User Stories and Acceptance Criteria to demonstrate how different people mean different things when they say the same things. It clicked for all of us that the “cost of change” is cheap when one has not implemented any code. As soon as code has been implemented the cost of change is “high”. I like how Dan Picket said that user stories and acceptance criteria serve the purpose of “getting a conversation started”. It is not about who “won” the right-wrong contest and blaming each other. I love the word “started” as it means you are opening a door to exchange ideas and explain yourself.
